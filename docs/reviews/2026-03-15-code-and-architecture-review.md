# Code And Architecture Review

Этот документ фиксировал проблемы исходного `teknofest`-сценария. После рефакторинга под multi-scene runtime findings ниже считаются закрытыми через новую архитектуру с `main.py`, `scenes/<competition>/<year>/`, wrapper CLI и YOLO export pipeline.

Текущий статус:

- legacy `teknofest.py` и старые `utils/*` удалены
- hardcoded paths, `pipenv` bootstrap и import-time execution удалены
- JSON-аннотации больше не являются primary export format
- вывод теперь идёт в `exports/<competition>_<year>_<timestamp>/`
- экспорт и split реализованы детерминированно через seed

Остаточный риск:

- Blender runtime нельзя полноценно проверить в этом окружении без установленного `blender`
- профиль `scenes/sauvc/2026/` пока шаблонный и требует реальной сцены

## Findings

### High

1. Жёстко заданные пути и bootstrap окружения ломают переносимость и запуск вне машины автора.

Файлы:

- `teknofest.py:7`
- `teknofest.py:20-26`

Проблема:

- Скрипт жёстко вызывает `pipenv --venv`, предполагает готовое окружение и руками добавляет `site-packages` из пути с зашитой версией `python3.10`.
- Затем он делает `os.chdir(base_dir)` в абсолютный путь, прошитый под конкретные машины.

Риск:

- запуск ломается в CI, на другой машине, в другом Python minor version и при любом отличии структуры путей;
- сценарий нельзя переиспользовать как библиотеку или вызывать параметризованно.

Минимальное исправление:

- вычислять корень проекта от `Path(__file__).resolve().parent`;
- убрать `os.chdir`;
- не инжектить `site-packages` вручную;
- запускать Blender с уже подготовленным окружением.

2. Запись JSON неатомарна и может оставлять битый хвост или смешанные результаты при повторных запусках.

Файлы:

- `teknofest.py:81`
- `teknofest.py:225-227`
- `utils/SampleData.py:24-35`

Проблема:

- итоговый JSON пишется в файл, открытый как `r+`, без `seek(0)` и `truncate()`;
- чтение существующих данных и запись новых разделены между разными открытиями файла;
- логика продолжения генерации опирается на существующие `jpg`, а не на консистентность JSON.

Риск:

- при более коротком новом содержимом в файле остаются старые байты;
- результаты могут расходиться между `results/` и JSON;
- восстановление после частично неудачного запуска затруднено.

Минимальное исправление:

- писать JSON атомарно через временный файл и `replace`;
- восстанавливать состояние из единственного источника правды;
- явно разделить режимы `append` и `fresh run`.

3. Поиск валидного ракурса может зависнуть навсегда.

Файл:

- `teknofest.py:149-220`

Проблема:

- используется бесконечный `while 1`, который завершится только если найдётся кадр минимум с двумя валидными bbox.

Риск:

- любой неудачный конфиг, слишком жёсткие пороги или проблемная сцена превращают генерацию в вечный цикл без диагностик.

Минимальное исправление:

- ввести лимит попыток на кадр;
- при превышении лимита логировать причину и переходить к следующей итерации или падать с понятной ошибкой.

4. Конфигурируемое качество рендера фактически применяется после рендера, а не до него.

Файл:

- `utils/rendering.py:8-13`

Проблема:

- `bpy.context.scene.cycles.samples = NUM_SAMPLES` вызывается после `bpy.ops.render.render(write_still=True)`.

Риск:

- текущий кадр рендерится с предыдущими настройками samples;
- пользователь меняет `NUM_SAMPLES` в YAML, но не получает ожидаемого эффекта.

Минимальное исправление:

- выставлять `cycles.samples` до вызова рендера.

### Medium

5. Вращение камеры реализовано через создание и удаление временного объекта.

Файл:

- `utils/setup_camera.py:4-18`

Проблема:

- `rotate_camera` создаёт `Camera.001`, переключает активную камеру и затем удаляет временный объект.

Риск:

- лишние side effects в сцене;
- зависимость от имени `Camera.001`;
- деградация производительности на длинных сериях.

Минимальное исправление:

- крутить существующую камеру напрямую без `bpy.ops.object.camera_add()`.

6. Импорт `utils/make_grid.py` имеет побочный эффект.

Файл:

- `utils/make_grid.py:48`

Проблема:

- модуль печатает результат `make_grid(...)` прямо при импорте.

Риск:

- шум в stdout;
- неожиданные side effects при повторном использовании модуля и в тестах.

Минимальное исправление:

- удалить `print(...)` или спрятать его под `if __name__ == "__main__":`.

7. Создание mesh для расчёта bbox потенциально течёт по памяти.

Файл:

- `utils/bboxes.py:63-65`
- `utils/bboxes.py:102`

Проблема:

- `me_ob.to_mesh(...)` создаёт mesh-копию, но её освобождение закомментировано.

Риск:

- при длинных прогонах Blender-сессия может накапливать мусор и деградировать по памяти.

Минимальное исправление:

- корректно освобождать временный mesh после расчёта bbox.

8. Порядок объектов нестабилен, а воспроизводимость отсутствует.

Файлы:

- `teknofest.py:83`
- `utils/objects_in_view.py:136-150`

Проблема:

- объекты для перемещения собираются в `set`, затем перечисляются через `enumerate`;
- random seed не фиксируется.

Риск:

- одинаковый запуск нельзя воспроизвести;
- отладка поведения и сравнение датасетов усложняются.

Минимальное исправление:

- использовать упорядоченную коллекцию объектов;
- добавить опциональный seed в конфиг.

### Low

9. Модели данных используются непоследовательно и только на границе JSON.

Файлы:

- `utils/SampleData.py:5-21`
- `teknofest.py:203-218`

Проблема:

- Pydantic-модели есть, но объект `Bbox` не используется при сборке результатов;
- код оперирует словарями вперемешку с моделями.

Риск:

- типизация не даёт практической пользы;
- правила сериализации и валидации разнесены по коду.

Минимальное исправление:

- собирать результаты через модели/датаклассы и сериализовать один раз на выходе.

10. Скрипт делает слишком много работы на верхнем уровне модуля.

Файл:

- `teknofest.py:1-227`

Проблема:

- при импорте выполняются bootstrap, загрузка конфига, инициализация путей и артефактов, чтение JSON и запуск основной логики.

Риск:

- модуль трудно тестировать;
- невозможно безопасно переиспользовать отдельные части.

Минимальное исправление:

- вынести orchestration в `main()` и разбить этапы на функции.

## Architecture Review

Сейчас архитектура организована вокруг одного procedural script:

- bootstrap окружения;
- загрузка config;
- чтение/подготовка scene state;
- генерация вариаций сцены;
- отбор видимых объектов;
- построение bbox;
- рендер;
- сериализация датасета.

Главный архитектурный долг:

- нет явной границы между scene-specific knowledge и generic pipeline;
- нет единого объекта конфигурации;
- нет явной модели job/run;
- нет CLI и интерфейса для добавления новых сцен;
- логика воспроизводимости, ошибок и состояния run не оформлена.

Из-за этого текущий код сложно:

- запускать на другой машине;
- расширять на другой `.blend`;
- тестировать без Blender;
- отлаживать при деградации качества датасета.

## Refactoring Backlog

### Phase 1: Stabilize Current Workflow

Status: completed in the multi-scene refactor.

- Убрать жёсткие пути и `os.chdir`, вычислять проектные пути от `__file__`.
- Вынести код из верхнего уровня в `main()`.
- Исправить порядок установки `cycles.samples`.
- Переписать запись JSON на атомарную.
- Добавить лимит попыток поиска камеры и понятные сообщения об ошибках.
- Удалить импортные side effects из `utils/make_grid.py`.
- Освобождать временные mesh-объекты после bbox-расчёта.
- Добавить опциональный seed в конфиг.

### Phase 2: Separate Pipeline From Blender Scene Details

Status: completed for the current config-driven scope.

- Ввести типизированный объект конфигурации для `render_settings`, labels и scene bindings.
- Разделить код на подсистемы:
  - scene adapter
  - sampling/orchestration
  - visibility filtering
  - annotation writer
  - renderer
- Убрать хардкод имён Blender-объектов и коллекций из алгоритмов в конфиг/scene profile.
- Сделать единый объект результата кадра вместо ручной сборки словарей.

### Phase 3: Prepare For Multiple Scenarios

Status: partially completed.

- Введён единый CLI `main.py run --blender ... --config ...`.
- Generic pipeline отделён от dataset-specific profiles через `scenes/<competition>/<year>/`.
- Добавлены минимальные unit-тесты для config resolution и YOLO export.
- Dry-run режим пока не реализован.

## Suggested Acceptance Criteria For Refactoring

- Скрипт запускается без ручной правки абсолютных путей.
- Один и тот же seed даёт воспроизводимый split и порядок sampling decisions.
- Аннотации пишутся в `manifest.json` атомарно, а финальный export не зависит от legacy JSON.
- Неудачная сцена завершается по `max_camera_attempts`, а не висит бесконечно.
- Cycles samples применяются до рендера.
- Добавление новой сцены делается через новый профиль в `scenes/<competition>/<year>/`.
